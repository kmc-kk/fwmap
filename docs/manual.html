fwmap Manual

fwmap is a local CLI tool for embedded firmware size analysis. It reads an ELF file and an optional GNU ld style map file, then generates a single HTML report that highlights ROM/RAM usage, large symbols, object contributions, and differences from a previous build.

Overview

This prototype is intended for fast local verification. It focuses on practical questions: which sections consume ROM or RAM, which symbols are largest, which object files contribute most, and what increased compared with the previous build.

Supported Scope

  • ELF32 / ELF64
  • Little-endian ELF as the main target
  • GNU ld style map files
  • Single HTML report output
  • Optional diff against a previous build
  • Fixed-threshold warnings

Current limitations include missing linker script parsing, no demangling, heuristic ROM/RAM estimation, and partial support for less common map variants.

Installation and Build

Requirements:

  • Rust toolchain
  • Windows, Linux, or macOS
cargo build
cargo test

Basic Usage

Analyze an ELF only

fwmap analyze --elf build/app.elf

If --out is omitted, the report is written to fwmap_report.html.

Analyze an ELF with a map file

fwmap analyze --elf build/app.elf --map build/app.map --out report.html

Analyze with previous-build diff

fwmap analyze --elf build/app.elf --map build/app.map --prev-elf prev/app.elf --prev-map prev/app.map --out report.html

CLI Options

OptionRequiredDescription
--elf <path>YesCurrent ELF file
--map <path>NoCurrent GNU ld map file
--prev-elf <path>NoPrevious ELF file for diff
--prev-map <path>NoPrevious map file for diff
--out <path>NoOutput HTML path
--helpNoShow help

Output

fwmap produces a CLI summary on standard output and one HTML report file. The report contains Header, Overview, Warnings, Memory Summary, Section Breakdown, Top Symbols, Top Object Contributions, Diff, and Footer sections.

Reading the Report

Overview

Shows binary path, architecture, ELF class, endian, total section count, ROM total, RAM total, warning count, and optional ROM/RAM diff.

Warnings

  • ROM usage over 85%
  • RAM usage over 85%
  • Large symbol at or above 4 KiB
  • .data growth over 5%
  • .bss growth over 5%
  • Large symbol growth spike in diff
  • Non-fatal map parse issues

Memory Summary

Sections are sorted by size and classified as ROM, RAM, or Other. ROM includes .text, .rodata, and read-only allocated sections. RAM includes .data, .bss, and writable allocated sections.

Section Breakdown

Shows section name, address, size, and flags.

Top Symbols

Shows up to 50 largest symbols from the ELF symbol table with section and object information when available.

Top Object Contributions

Available when a map file is provided. This is useful for spotting which object files are dominating image size.

Diff

When previous inputs are provided, the report compares ROM, RAM, section sizes, symbol sizes, object contributions, and added or removed symbols.

Error Handling

Typical user-facing errors include missing ELF/map files, non-ELF input, unsupported ELF metadata, and write failures for the HTML report. Map parse failures are retained as warnings whenever partial output is still possible.

Internal Structure

  • cli: argument parsing and execution flow
  • ingest: ELF and map readers
  • model: shared data structures
  • analyze: summaries, diffs, warnings
  • render: CLI summary and HTML output

Typical Workflows

Quick local size check

fwmap analyze --elf build/app.elf

Investigate object growth

fwmap analyze --elf build/app.elf --map build/app.map

Compare two builds

fwmap analyze --elf current/app.elf --map current/app.map --prev-elf previous/app.elf --prev-map previous/app.map --out diff_report.html

Known Constraints

  • Some GNU ld map variants may only be partially parsed
  • Archive/member aggregation is based on map line patterns
  • Exact memory-region usage is not available without linker script interpretation
  • Dynamic symbols and DWARF-based source attribution are out of scope
