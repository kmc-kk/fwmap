fwmap マニュアル

fwmap は、組込みファームウェアの ELF、GNU ld 系 map、linker script を解析し、サイズ・差分・memory region 配置を可視化するローカル CLI ツールです。

この版では、JSON 出力、CI 向け要約、しきい値指定、warning ベースの終了コード制御、外部ルール設定、C++ symbol demangle まで含めて説明します。

1. 何ができるか

  • ROM / RAM の概算を確認する
  • 大きい symbol を見つける
  • どの object file がサイズに効いているか把握する
  • 前回ビルドとの差分を分類付きで確認する
  • FLASH / RAM などの region 使用率を確認する
  • CI で機械判定できる JSON と短い要約を出す
  • warning を rule code 単位で追跡する
  • C++ symbol を読みやすい名前で表示する

2. 事前に用意するファイル

project/
  build/
    app.elf
    app.map
  linker/
    app.ld
  prev/
    app.elf
    app.map

最低限必要なのは app.elf です。object 寄与を見るなら app.map、region 配置を見るなら app.ld、回帰比較には prev/app.elf と prev/app.map を使います。

3. 実際の使用手順

手順 1: ELF だけで全体確認

fwmap analyze --elf build/app.elf

まず section 数、ROM / RAM 概算、大きい symbol、基本 warning を確認します。map が無い環境でも使えます。

手順 2: map を付けて object 寄与を見る

fwmap analyze --elf build/app.elf --map build/app.map --out build/fwmap_report.html

Top Object Contributions と Top Symbols を見比べます。たとえば drivers/net.o と g_rx_ring が上位に出れば、受信バッファ由来の増加を疑えます。

手順 3: linker script を付けて region 配置を見る

fwmap analyze --elf build/app.elf --map build/app.map --lds linker/app.ld --out build/fwmap_region_report.html

Memory Regions Overview で FLASH / RAM の使用率を見て、Region Sections で各 region にどの section が載っているかを確認します。

たとえば FLASH が 91% なら危険域です。SECTION_REGION_MISMATCH が出ていれば、linker script の想定配置と実アドレスがずれている可能性があります。

手順 4: 前回ビルドとの差分を見る

fwmap analyze --elf build/app.elf --map build/app.map --prev-elf prev/app.elf --prev-map prev/app.map --out build/fwmap_diff_report.html
ELF: build/app.elf
ROM: 65536 bytes (64.00 KiB) | RAM: 16384 bytes (16.00 KiB) | Sections: 30 | Symbols: 240 | Warnings: 2
ROM: +3072 bytes
RAM: +1024 bytes
Diff counts: sections +1 / -0 / ↑4 / ↓1, symbols +3 / -1 / ↑7 / ↓2
Top growth symbol: app_tls_buffer (+1024)
Top growth object: middleware/tls.o (+1536)
Report: build/fwmap_diff_report.html

この場合は app_tls_buffer と middleware/tls.o を起点に変更を追います。

手順 5: warning を詳しく見る

fwmap analyze --elf build/app.elf --map build/app.map --lds linker/app.ld --verbose
Warnings:
  [analyze:REGION_THRESHOLD] Region FLASH usage exceeded 85% (91.2%)
  [analyze:REGION_LOW_FREE] Region RAM free space is low (2048 bytes (2.00 KiB))
  [analyze:LARGE_SYMBOL] Large symbol detected: g_rx_ring (12288 bytes (12.00 KiB))
  [map:MAP_LINE_SKIPPED] Skipped unparsed map line 120: COMMON ...

source で発生元、code で warning の種類、related で対象 region / section / symbol を確認できます。

手順 6: JSON を出力して CI やスクリプトで使う

fwmap analyze --elf build/app.elf --map build/app.map --lds linker/app.ld --report-json build/fwmap_report.json

JSON には binary、memory summary、warnings、thresholds、top symbols、top object contributions、regions、diff 情報が含まれます。

手順 6.1: 外部ルール設定を読み込む

fwmap analyze --elf build/app.elf --map build/app.map --rules tests/fixtures/sample_rules.toml

TOML ルールファイルから threshold override と custom rule を読み込みます。壊れた設定や必須項目不足は解析前に明確なエラーになります。

手順 6.2: C++ symbol を demangle する

fwmap analyze --elf build/app.elf --map build/app.map --demangle=on

auto は Itanium ABI らしい名前だけを対象にし、on は積極的に変換を試し、off は raw 名のまま表示します。

手順 7: CI 向けの短い要約を出す

fwmap analyze --elf build/app.elf --map build/app.map --prev-elf prev/app.elf --prev-map prev/app.map --ci-format markdown --ci-out build/fwmap_ci.md
ROM: +12345 bytes
RAM: +2048 bytes
Warnings: 2
Errors: 1
Top section growth: .bss (+8192)
Top symbol growth: g_rx_ring (+4096)
Triggered rules: forbid-g-rx-ring(error), REGION_THRESHOLD(warn)

--ci-format では text、markdown、json を選べます。--ci-summary は text 形式の簡易指定としてそのまま使えます。

手順 8: warning が出たら失敗にする

fwmap analyze --elf build/app.elf --map build/app.map --fail-on-warning

error severity の rule が 1 件以上あれば終了コード 2 になります。--fail-on-warning 指定時に warn / info の warning があれば終了コード 1 になります。

手順 9: しきい値を調整する

fwmap analyze --elf build/app.elf --map build/app.map --threshold-rom 90 --threshold-ram 90 --threshold-region FLASH:92 --threshold-symbol-growth 8192

この例では ROM/RAM/FLASH/symbol growth の warning 発火条件を変更しています。

手順 10: 外部ルールファイルを書く

schema_version = 1

[thresholds]
rom_usage_warn = 0.90
ram_usage_warn = 0.88

[[rules]]
id = "flash-near-full"
kind = "region_usage"
region = "FLASH"
warn_if_greater_than = 0.92
severity = "warn"
message = "FLASH usage is above 92%"

[[rules]]
id = "tls-data-growth"
kind = "section_delta"
section = ".data"
warn_if_delta_bytes_gt = 2048
severity = "warn"
message = ".data increased by more than 2KB"

現行版で対応している kind は region_usage、section_delta、symbol_delta、symbol_match、object_match です。

4. CLI オプション

オプション必須説明
--elf <path>必須現在の ELF ファイル
--map <path>任意現在の GNU ld map ファイル
--lds <path>任意GNU ld linker script
--prev-elf <path>任意比較用の前回 ELF
--prev-map <path>任意比較用の前回 map
--out <path>任意HTML 出力先
--report-json <path>任意JSON 出力先
--rules <path>任意外部 TOML ルール設定
--demangle=auto|on|off任意C++ symbol demangle 制御
--ci-summary任意CI 向けの短い要約を表示
--ci-format <text|markdown|json>任意CI 要約の出力形式
--ci-out <path>任意CI 要約の出力先
--fail-on-warning任意warning があれば非 0 終了
--threshold-rom <percent>任意ROM warning しきい値
--threshold-ram <percent>任意RAM warning しきい値
--threshold-region <name:percent>任意region ごとの warning しきい値
--threshold-symbol-growth <bytes>任意symbol growth warning しきい値
--verbose任意warning 詳細を標準出力へ表示
--version任意バージョン表示
--help任意ヘルプ表示

5. 出力の種類

  • 標準出力: 通常サマリまたは CI 向け短縮サマリ
  • HTML: 人が読むための詳細レポート
  • JSON: 機械可読な固定 schema のレポート
  • CI summary: text / markdown / JSON の短い要約

JSON の top-level key は schema_version, binary, memory_summary, warnings, thresholds, regions, diff_summary, diff などです。top_symbols には raw 名の name と表示用の demangled_name が両方含まれます。

6. HTML レポートの見方

  1. Overview: 全体概要
  2. Warnings: 問題点と読み飛ばし情報
  3. Memory Summary: section ごとの ROM/RAM 集計
  4. Memory Regions Overview: region 使用率
  5. Region Sections: region 配下 section 一覧
  6. Section Breakdown: section 詳細
  7. Top Symbols: 大きい symbol
  8. Top Object Contributions: 大きい object
  9. Diff: 増減一覧と追加削除

warning 判定はルール単位で分離され、外部 TOML ルールも読み込めるため、同じ code を軸に CLI、HTML、JSON を横断して追跡しやすくなっています。

7. よくある確認パターン

RAM が急に増えた

Top Symbol Growth と Top Object Growth を見て、.bss / .data diff を確認します。

C++ symbol が読みにくい

--demangle=on を付けて再実行します。比較キーは raw symbol 名のままなので、差分の安定性は維持されます。

同じ warning が繰り返し出る

code と related を見て、同一 region なのか同一 symbol なのかを先に切り分けます。たとえば REGION_THRESHOLD と FLASH が固定なら linker script やコード配置、LARGE_SYMBOL と g_rx_ring が固定なら静的バッファ定義を優先して確認します。

チーム独自ルールを足したい

--rules で TOML を読み込みます。まずは region_usage や section_delta から始めると運用しやすいです。

FLASH が逼迫している

Memory Regions Overview の FLASH 使用率と Top Section Growth を確認します。

map が少し壊れている

解析は可能な範囲で継続し、読み飛ばした箇所を warning に残します。まず HTML が生成されるか、次に MAP_LINE_SKIPPED を確認してください。

8. 既知の制約

  • ELF は現在 SHT_SYMTAB を中心に参照
  • map は GNU ld の典型出力を優先
  • linker script は subset 対応で、完全構文対応ではない
  • region 使用量は linker script と ELF section address の組み合わせに依存する
  • demangle は現在 Itanium ABI 系の軽量対応
  • 外部ルール設定は TOML 固定で、対応 kind は現在の実装範囲に限られる
  • JSON schema は現時点で schema_version = 1

現行実装に合わせて、具体的な使用手順、外部ルール設定、demangle 運用を含めて更新したマニュアルです。
